ISSUE 4: TC-RPT-ADV-007 - BigDecimal Division Assertion Risky
================================================================================
File: ReportServiceAdvancedTest.java
Line: 185

BEFORE (direct assertEquals - scale-dependent):
  assertEquals(new BigDecimal("33.33"), result.getAvgOrderValue());

AFTER (compareTo - scale-independent):
  assertEquals(0, new BigDecimal("33.33").compareTo(result.getAvgOrderValue()),
      "avgOrderValue must be 33.33 with scale=2, HALF_UP rounding");
  
  // ⚠️ RISK: If service uses divide() without scale/rounding:
  // BigDecimal.divide(divisor) → ArithmeticException: Non-terminating decimal expansion
  // Must use: divide(divisor, 2, RoundingMode.HALF_UP)

IMPACT:
- Uses compareTo() instead of assertEquals() for BigDecimal comparison
- Scale-independent comparison (33.33 == 33.330)
- Documents risk of ArithmeticException if service doesn't specify scale/rounding
- Clear message about expected rounding mode
